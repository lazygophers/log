📜 行为准则#

本项目及其所有参与者都受我们的行为准则约束。通过参与，您需要遵守此准则。

🔄 开发流程#

我们使用 GitHub 来托管代码、跟踪问题和功能请求，以及接受拉取请求。

工作流程#

  1. Fork 仓库
  2. Clone 您的 fork 到本地
  3. 创建 从 master 分支创建功能分支
  4. 进行 您的更改
  5. 测试 在所有构建标签下进行彻底测试
  6. 提交 拉取请求

🚀 开始使用#

先决条件#

  • Go 1.21+ - 安装 Go
  • Git - 安装 Git
  • Make (可选但推荐)

本地开发设置#

# 1. 在 GitHub 上 Fork 仓库
# 2. Clone 您的 fork
git clone https://github.com/YOUR_USERNAME/log.git
cd log

# 3. 添加上游远程仓库
git remote add upstream https://github.com/lazygophers/log.git

# 4. 安装依赖
go mod tidy

# 5. 验证安装
make test-quick

环境设置#

# 设置 Go 环境（如果尚未设置）
export GOPATH=$HOME/go
export PATH=$PATH:$GOPATH/bin

# 可选：安装有用的工具
go install golang.org/x/tools/cmd/goimports@latest
go install honnef.co/go/tools/cmd/staticcheck@latest

📨 拉取请求流程#

提交前#

  1. 搜索 现有 PR 以避免重复
  2. 测试 您的更改在所有构建配置下
  3. 记录 任何破坏性更改
  4. 更新 相关文档
  5. 添加 新功能的测试

PR 检查清单#

  • 代码质量

    • 代码遵循项目风格指南
    • 没有新的 lint 警告
    • 正确的错误处理
    • 高效的算法和数据结构

  • 测试

    • 所有现有测试通过：make test
    • 为新功能添加了新测试
    • 测试覆盖率保持或提高
    • 所有构建标签已测试：make test-all

  • 文档

    • 代码有适当的注释
    • API 文档已更新（如果需要）
    • README 已更新（如果需要）
    • 多语言文档已更新（如果是面向用户的）

  • 构建兼容性

    • 默认模式：go build
    • 调试模式：go build -tags debug
    • 发布模式：go build -tags release
    • 丢弃模式：go build -tags discard
    • 组合模式已测试

📏 编码标准#

Go 风格指南#

我们遵循标准的 Go 风格指南，并有一些补充：

// ✅ Good
func (l *Logger) Info(v ...any) {
    if l.level > InfoLevel {
        return
    }
    l.log(InfoLevel, fmt.Sprint(v...))
}

// ❌ Bad
func (l *Logger) Info(v ...any){
    if l.level>InfoLevel{
        return
    }
    l.log(InfoLevel,fmt.Sprint(v...))
}

命名约定#

  • 包: 简短、小写、尽可能使用单个单词
  • 函数: 驼峰命名法，具有描述性
  • 变量: 本地变量使用 camelCase，导出变量使用 CamelCase
  • 常量: 导出常量使用 CamelCase，非导出常量使用 camelCase
  • 接口: 通常以 "er" 结尾（例如 Writer、Formatter）

代码组织#

project/ ├── docs/ # 多语言文档 ├── .github/ # GitHub 模板和工作流 ├── logger.go # 主要日志记录器实现 ├── entry.go # 日志条目结构 ├── level.go # 日志级别 ├── formatter.go # 日志格式化 ├── output.go # 输出管理 └── *_test.go # 与源代码共存的测试

错误处理#

// ✅ 推荐：返回错误，不要 panic
func NewLogger(config Config) (*Logger, error) {
    if err := config.Validate(); err != nil {
        return nil, fmt.Errorf("invalid config: %w", err)
    }
    return &Logger{...}, nil
}

// ❌ 避免：在库代码中使用 panic
func NewLogger(config Config) *Logger {
    if err := config.Validate(); err != nil {
        panic(err) // 不要这样做
    }
    return &Logger{...}
}

覆盖率要求#

  • 最低要求: 新代码覆盖率 90%
  • 目标: 总体覆盖率 95%+
  • 所有构建标签必须保持覆盖率
  • 使用 make coverage-all 进行验证

代码文档#

  • 所有导出的函数必须有清晰的注释
  • 复杂算法需要解释
  • 示例用于非平凡用法
  • 线程安全说明（如适用）
// SetLevel sets the minimum logging level.
// Logs below this level will be ignored.
// This method is thread-safe.
//
// Example:
//   logger.SetLevel(log.InfoLevel)
//   logger.Debug("ignored")  // Won't output
//   logger.Info("visible")   // Will output
func (l *Logger) SetLevel(level Level) *Logger {
    l.mu.Lock()
    defer l.mu.Unlock()
    l.level = level
    return l
}

🚀 性能考虑#

基准测试#

始终对性能敏感的更改进行基准测试：

# 运行基准测试
go test -bench=. -benchmem

# 比较前后性能
go test -bench=. -benchmem > before.txt
# 进行更改
go test -bench=. -benchmem > after.txt
benchcmp before.txt after.txt

性能指南#

  • 最小化 热路径中的内存分配
  • 使用对象池 用于频繁创建的对象
  • 早期返回 对于禁用的日志级别
  • 避免反射 在性能关键代码中
  • 在优化前进行性能分析

内存管理#

// ✅ 推荐：使用对象池
var entryPool = sync.Pool{
    New: func() interface{} {
        return &Entry{}
    },
}

func getEntry() *Entry {
    return entryPool.Get().(*Entry)
}

func putEntry(e *Entry) {
    e.Reset()
    entryPool.Put(e)
}

🔒 安全指南#

敏感数据#

  • 永远不要记录密码、令牌或敏感数据
  • 清理日志消息中的用户输入
  • 避免记录整个请求/响应体
  • 使用结构化日志记录以获得更好的控制
// ✅ 推荐
logger.Info("User login attempt", "user_id", userID, "ip", clientIP)

// ❌ 避免
logger.Infof("User login: %+v", userRequest) // 可能包含密码

依赖项#

  • 保持依赖项最新
  • 仔细审查新的依赖项
  • 最小化外部依赖项
  • 使用 go mod verify 检查完整性
